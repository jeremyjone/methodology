# 一、经典头脑风暴（Brainstorming）

## 1.1 什么是头脑风暴

头脑风暴（Brainstorming）是一种集思广益的群体创意方法，通过组织专家、学者、创意人员等以会议形式共同围绕明确议题进行讨论，借助与会者的群体智慧，互相启发激励，引发创造性设想的连锁反应，以产生和发展出众多创意构想。

**创始人**：亚历克斯·奥斯本（Alex F. Osborn），美国BBDO广告公司合伙人。1939年开始发展群体创意方法，1942年在《How to Think Up》中首次以专著形式提及该方法，1948年在《Your Creative Power》第33章"How to Organize a Squad to Create Ideas"中详细阐述，1953年在《Applied Imagination》中完整系统化。

**别称**：智力激励法、脑力激荡法、畅谈会、集思法、BS法。

**核心原理**：在群体决策中，由于群体成员心理相互作用影响，易屈于权威或大多数人意见，形成所谓的"群体思维"（Groupthink），削弱了群体的批判精神和创造力，损害了决策质量。头脑风暴通过一系列规则设计，保证群体决策的创造性。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 1. Osborn《Applied Imagination》、2. Osborn《Your Creative Power》、3.《Brainstorming》- Wikipedia、4.《17种高效头脑风暴方法》、5.《创意思考之五大头脑风暴法》

## 1.2 四大核心原则

### 原则一：自由畅想

欢迎各抒己见，自由鸣放，创造一种自由、活跃的气氛，激发参加者提出各种荒诞的想法，使参与者思想放松。提倡自由发言，畅所欲言，任意思考，任意想象，尽量发挥，主意越新、越怪越好。

### 原则二：延迟批判（暂缓评价）

不做任何有关优缺点的评价，不允许提出质疑或批评。禁止说"这个想法不行"、"这不现实"之类的话。所有评判都要留到会后。

### 原则三：以量求质

追求数量，意见越多，产生好意见的可能性越大。奥斯本研究显示，在产生设想阶段，越是增加设想的数量，就越有可能获得富有实用价值的设想。

### 原则四：综合改善

鼓励巧妙地利用和改善他人的设想。每个参与者都要从他人的设想中激励自己，得到启示，或补充他人的设想，或将他人的若干设想综合起来提出新的设想。

## 1.3 标准操作流程

**第一步：准备阶段**

- 明确要解决的问题或议题
- 确定与会人数（5-10人为宜）
- 选择主持人（应善于启发且自身思维敏捷）
- 提前通知与会者议题，并附送必要的说明资料

**第二步：说明规则**

- 宣布四大原则
- 强调禁止批评
- 设定时间限制
- 说明会议目标

**第三步：热身（可选）**

- 主持人先抛出一个简单有趣的开放问题
- 让大家适应无批评的氛围
- 约5-10分钟

**第四步：自由畅想**

- 围绕议题提出各种想法
- 主持人鼓励所有人参与
- 记录员记录所有想法（不加评判）
- 营造热烈、轻松的气氛

**第五步：整理与评估**

- 将所有想法整理分类，编出一览表
- 逐一质疑，分析可行性
- 挑出最有希望的见解
- 审查并形成最终方案

## 1.4 适用场景

- 解决复杂问题，但尚未形成具体方案
- 需要产生大量创意和点子
- 团队需要打破固有思维
- 项目初期，方向尚未明确
- 产品功能设计、营销策略、名称征集等创意类工作

## 1.5 不适用场景

- 已进入执行阶段的方案优化（更适合其他方法）
- 需要精确数据计算的决策
- 时间极度紧迫（头脑风暴需要足够的时间发散）
- 团队成员少于3人

## 1.6 注意事项

1. **人数控制**：5-10人为宜。人数过少创意有限，人数过多则难以保证每个人充分发言。

2. **专业搭配**：尽量避免行家过多。行家过多容易过早评价，形成权威压制，难以形成自由奔放的讨论气氛。

3. **主持人角色**：主持人一般不发表意见，以免影响会议的自由气氛。要善于启发、引导，保持会议热烈气氛。

4. **时间控制**：单次时长建议45-60分钟。时间太短无法充分发散，时间太长则疲劳且效率下降。

5. **记录完整**：所有想法原样记录，会后不追责。

### 1.7 示例

**议题：如何提升产品用户留存率？**

### 自由畅想阶段实录（节选）

**参与者A**：做一个签到功能，每天签到送积分！

**参与者B**：（不对A的想法做评价）我觉得可以做一个虚拟宠物，用户养着它，走了就死了。

**参与者C**：宠物太常见了……但"死了"这个概念有意思！我觉得可以做一个"植物养成"，每天浇水施肥。

**参与者D**：社交！做一个用户之间的互动社区，让大家聊起来。

**参与者E**：游戏化成就系统，完成任务解锁勋章。

**参与者F**：推送太频繁了，我建议减少推送，只在用户可能需要的时候才推送。

**参与者G**：会员制，不付费的用户功能受限，但又不影响核心体验。

**参与者H**：接入AI助手，个性化推荐内容，用户觉得"懂我"就会留下。

**参与者I**：线下活动，用户在线下认识了就更容易线上留存。

**参与者J**：做一个小程序，轻量级，不用下载APP。

（以上想法在畅想阶段全部记录，无一被否定）

### 会后评估阶段

将所有想法分类：

| 类别 | 想法 |
|------|------|
| 功能类 | 签到积分、虚拟宠物/植物养成、成就系统、AI个性化推荐 |
| 社交类 | 用户社区、线下活动 |
| 运营类 | 减少推送频率 |
| 商业类 | 会员制 |
| 技术类 | 小程序轻量化 |

评估可行性后，最终保留：签到积分 + AI个性化推荐 + 会员制的组合方案。

# 二、质疑头脑风暴（Reverse Brainstorming）

## 2.1 什么是质疑头脑风暴

质疑头脑风暴（Reverse Brainstorming）又称反向头脑风暴、逆向头脑风暴（Negative Brainstorming / Problem Reversal），是将典型头脑风暴过程颠倒过来的方法。不是问"我们如何做成功"，而是问"我们会怎么失败"。

**核心思想**：一旦知道什么是无效的，就拥有了实现目标所需的方法。

**适用场景**：已有初步方案，需要找出漏洞；或者问题方向不明，需要先排除死路。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 6. Michalko《Thinkertoys》、7. VanGundy《Techniques of Structured Problem Solving》

## 2.2 操作流程

1. 将问题反转成"如何搞砸/破坏"
2. 围绕"如何搞砸"列举所有可能
3. 将失败原因反过来，推导出成功路径

### 2.3 示例

**议题：如何改善客户服务？**

### 第一步反转问题

原问题：如何改善客户服务？

反向问题：我们如何使客户服务糟糕到让每个客户都流失？

### 第二步列举"搞砸"的方式

- 让客户等待超过10分钟才有人接电话
- 客服态度冷漠，用模板回复，不解决实际问题
- 每次转接，客户要重复描述问题3遍
- 承诺24小时回复但永远不回复
- 解决问题后不跟进，客户不知道问题是否真正解决
- 客服不具备专业知识，无法回答基本问题
- 客户投诉后石沉大海，没有任何反馈
- 不同的客服给不同的答案，没有统一标准
- 让客户在APP里找联系方式找半天找不到
- 深夜/节假日完全找不到人工客服

### 第三步反向推导成功路径

| 失败做法 | 成功路径 |
|----------|----------|
| 等待超10分钟 | 30秒内接听，或提供即时在线客服 |
| 模板回复 | 培训客服个性化回复，解决实际问题 |
| 转接重复描述 | 建立统一工单系统，客服可查看历史记录 |
| 承诺不兑现 | 设置SLA，超时自动升级处理 |
| 不跟进 | 建立闭环机制，100%回访确认解决 |
| 专业知识不足 | 定期培训，建立知识库 |
| 投诉无反馈 | 24小时内必回复，重要投诉升级处理 |
| 答案不统一 | 制定标准答案库，定期更新 |
| 联系渠道难找 | 简化入口，APP/网站明显位置放联系方式 |
| 节假日无客服 | 提供AI客服+值班人工组合 |

# 三、脑力书写法 6-3-5（Brainwriting）

## 3.1 什么是脑力书写法

脑力书写法（Brainwriting）是一种"静默版"的头脑风暴，参与者不开口说话，而是将想法写下来，然后匿名传递。

**创始人**：德国学者 Bernd Rohrbach，于1969年在德国杂志 *Absatzwirtschaft* 上首次发表"Methode 635"。

解决了以下痛点：

- 社恐人士不爱发言
- 口头讨论容易被外向的人主导
- 书面形式可以产生更多想法（没有说完就忘的顾虑）
- 匿名性能减少顾虑，更大胆

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 8. Rohrbach "Kreativ nach Regeln – Methode 635"、9. VanGundy "Brain Writing for New Product Ideas"

## 3.2 6-3-5 规则详解

**6** = 六人一组

**3** = 每轮每人写下3个想法

**5** = 每轮持续5分钟

**6轮** = 总计30分钟，共产出 6人 × 3个想法 × 6轮 = **108个想法**

## 3.3 操作流程

1. **分组**：每组6人，安排座位（围桌或排排坐）
2. **发工作表**：每人一张工作表，表格有6个空格（对应6轮）
3. **第一轮**：在5分钟内写下3个想法，将工作表传递给右手边的人
4. **后续轮次**：每人收到左边传来的工作表后，在5分钟内阅读已有想法，然后写下3个新的或受启发延伸的想法，将表继续传递
5. **循环6轮**
6. **汇总**：所有工作表收集后，主持人朗读所有想法，集体去重、分类、评估

## 3.4 工作表示例

```
┌────┬──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│轮次│ 第1轮    │ 第2轮    │ 第3轮    │ 第4轮    │ 第5轮    │ 第6轮    │
├────┼──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│想法1│          │          │          │          │          │          │
├────┼──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│想法2│          │          │          │          │          │          │
├────┼──────────┼──────────┼──────────┼──────────┼──────────┼──────────┤
│想法3│          │          │          │          │          │          │
└────┴──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
```

### 3.5 与头脑风暴的对比

| 维度 | 口头头脑风暴 | 脑力书写6-3-5 |
|------|-------------|--------------|
| 媒介 | 说话 | 书写 |
| 互动性 | 高（当场碰撞） | 低（顺序传递） |
| 社恐友好度 | 低 | 高 |
| 想法数量 | 中等 | 大量（108个/组） |
| 匿名性 | 低 | 高 |
| 适合想法类型 | 快速反应、即时碰撞 | 深思熟虑、延伸联想 |
| 适用人数 | 5-15人 | 6的倍数 |

### 3.6 示例

**议题：新的会员权益体系如何设计？**

### 第一轮（5分钟）

**参与者A**的想法：

1. 每月免费配送权益
2. 专属客服通道
3. 生日当月双倍积分

**参与者B**的想法：

1. 专属折扣日
2. 新品优先购买权
3. 线下活动优先参与

（传递……）

### 第二轮（收到左边传来的表，继续写）

参与者A收到参与者F的表，看到：

- 积分兑换星巴克咖啡
- 机场贵宾室体验
- 电影票折扣

参与者A受启发，追加：

1. 与航空公司合作积分互通
2. 专属旅游优惠
3. 高端体检套餐兑换

### 最终汇总

6人 × 3个想法 × 6轮 = 108个原始想法，去重后约60个有效想法，归纳为：

- 折扣类（专属折扣日、积分抵现、电影票优惠）
- 体验类（机场贵宾室、线下活动、高端体检）
- 服务类（专属客服、优先购买、新品试用）
- 跨界类（航空积分互通、星巴克积分兑换、旅游优惠）

最终方案：三选二权益自由组合。

# 四、触发因子头脑风暴（Trigger Brainstorming）

### 4.1 什么是触发因子头脑风暴

触发因子头脑风暴（Trigger Brainstorming），又称随机刺激法（Random Stimuli Technique），通过随机词汇、图片、声音或其他刺激物，打破思维定式，触发跨界联想。这种方法利用"强制联想"的原理，让完全无关的事物产生碰撞，激发出乎意料的创新。

**理论来源**：Charles S. Whiting 于1958年在《Creative Thinking》中提出"触发法"的概念。爱德华·德·博诺（Edward de Bono）在1970年《Lateral Thinking》中系统化阐述了"随机输入法"（Random Input）技术，是类似概念的另一经典来源。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 10. Whiting《Creative Thinking》、11. De Bono《Lateral Thinking》

### 4.2 触发因子的类型

### 随机词汇法

准备一袋写有随机名词的卡片（如：石头、蝴蝶、钢琴、雨伞、咖啡、外星人）。讨论时随机抽一张，围绕该词与议题进行强制关联。

### 随机图片法

准备大量与议题无关的图片，参与者随机抽取，用图片激发灵感。

### 随机声音法

播放随机的声音（自然音、城市音、抽象音等），让参与者围绕"如果这个声音代表某种体验，会是什么"来联想。

### 随机句式法

准备若干开放句式，如：

- "如果这个问题是一个季节，它是什么季节？"
- "如果这个问题是一个人，它是什么性格的人？"
- "这个问题最不可能的解决方案是什么？"

### 4.3 操作流程

1. **准备触发物**：准备好随机词汇卡/图片卡/声音素材
2. **明确议题**：清晰说明要解决的问题
3. **抽取触发物**：随机抽取一个触发因子
4. **强制联想**：围绕触发因子与议题进行强制关联
5. **记录灵感**：写下所有由此产生的想法
6. **重复多轮**：更换触发物，继续联想
7. **汇总评估**

### 4.4 示例

**议题：如何让用户爱上我们的APP？**

### 触发物抽取：随机词汇卡——"蝴蝶"

### 强制联想

**Q：这个"蝴蝶"和"让用户爱上APP"有什么关系？**

联想链：

- 蝴蝶是美丽的 → 界面要美得像蝴蝶的翅膀
- 蝴蝶是轻盈的 → APP要轻快、不卡顿、响应快
- 蝴蝶会授粉 → 用户帮我们传播（口碑推荐）
- 蝴蝶的生命周期 → 用户的生命周期管理
- 蝴蝶是群居的 → 用户社区
- 破茧成蝶 → 用户从新手到高手的成长体系
- 蝴蝶迁徙 → 用户的跨平台迁移

### 产出想法

1. **界面美化**：推出蝴蝶主题皮肤，高级感
2. **性能优化**：打造"轻盈如蝶"的体验，加载时间<1秒
3. **推荐奖励**：用户推荐好友，双双获得"蝴蝶徽章"
4. **成长体系**：用户等级体系——毛毛虫→蛹→蝴蝶
5. **社区建设**：用户之间互称"蝶友"，建立社群
6. **跨端同步**：云端数据同步，用户走到哪都能"飞"

# 五、游戏化风暴法（Gamestorming）

### 5.1 什么是游戏化风暴法

游戏化风暴法（Gamestorming）通过游戏的形式来表达解决问题的想法，旨在提高团队成员的参与积极性和协作质量。它透过游戏技巧的角度发现创新点、突破点，从而扩宽思维。

**创始人**：Dave Gray、Sunni Brown、James Macanufo 三人共同创立，2010年出版《Gamestorming: A Playbook for Innovators, Rulebreakers, and Changemakers》（O'Reilly Media）。

**核心理念**：玩的时候人是没有心理防御的，更容易说出真实想法。

**核心结构**：将工作坊活动结构化为**开场（Opening）→ 探索（Exploring）→ 收敛（Closing）**三个阶段，包含80余种具体的"游戏"工具。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 12. Gray, Brown & Macanufo《Gamestorming》、13.《Gamestorming》官网

### 5.2 经典游戏化风暴游戏

以下游戏均收录于 Gray/Brown/Macanufo《Gamestorming》原书或 gamestorming.com 官方游戏库。

### 游戏一：Empathy Map（共情地图）

在大白纸上画一张分为四象限（Says / Thinks / Does / Feels，加上 Pains 与 Gains 两个补充区）的人物画像，团队围绕一个特定用户角色，把假设填入各象限。强迫团队跳出"我以为用户……"的视角，对齐用户真实处境。

### 游戏二：Speed Boat（快艇）

白板上画一艘船，船头是目标，船底画"锚"代表阻碍。每人用便利贴写出阻碍团队/产品前进的"锚"，越大的锚代表阻力越大。视觉化地暴露障碍并据此排序。

### 游戏三：Cover Story（封面故事）

请团队想象 2 年后某权威媒体（如《时代》《财经》）以本产品/项目作为封面故事，画出杂志封面、大标题、引文、专栏、新闻短讯、图说等。倒推出团队真正期待的成功画面。

### 游戏四：Bodystorming（身体风暴）

参与者用身体动作、走位、道具来"演"出用户场景或产品体验，而非只是讨论。通过具身认知暴露纸面方案中看不见的卡点。

### 游戏五：Yes, and...（即兴接龙，源自即兴戏剧）

一个人提出一个想法，下一个人必须说"是的，而且……"继续延伸。不许说"但是"或"不行"，强制产出大量衍生想法。该规则由即兴戏剧大师 Keith Johnstone 与 Del Close 系统化，被《Gamestorming》列为通用规则之一。

### 5.3 操作要点

1. **破冰**：先玩一个简单的游戏让大家放松
2. **规则清晰**：每个游戏有明确的规则和时间
3. **记录**：安排专人记录游戏中的洞见
4. **复盘**：游戏结束后，讨论游戏过程中产生的想法

### 5.4 示例

**议题：如何提升团队协作效率？**

### 游戏选择：Yes, and...（即兴接龙）

### 第一轮

**参与者A**：我觉得我们应该每天站会。

**参与者B**：是的，而且我们可以站着开完再买咖啡！

**参与者C**：是的，而且咖啡可以由迟到的人买单！

**参与者D**：是的，而且迟到的人要学一个笑话讲给大家！

**参与者E**：是的，而且学不好笑的话……多学一个！

### 第二轮

**参与者A**：我觉得可以用项目管理软件追踪任务。

**参与者B**：是的，而且可以设置自动提醒，不完成就罚唱歌！

**参与者C**：是的，而且唱歌要录屏发群里！

**参与者D**：是的，而且如果唱得好听，团队基金奖励他！

### 复盘讨论

从这些荒诞的延伸中提取可行想法：

- ✅ 每天站会 → 每日简短的站会对齐会
- ✅ 自动提醒 → 任务管理软件的提醒功能
- ✅ 迟到罚则 → 迟到惩罚机制（改为奖励准时）
- ✅ 唱歌表演 → 团队破冰/展示环节

# 六、窃取风暴法（Cheatstorming）

### 6.1 什么是窃取风暴法

窃取风暴法（Cheatstorming）将已有的想法或他人的创意集合起来，作为输入或刺激，通过拼接、混合、夹杂、变异、演化等方法，得到新的创意。

**核心理念**：不浪费之前的数据信息。相当于"站在巨人的肩膀上"的认知策略。其哲学根基可追溯到 Austin Kleon 的"像艺术家一样偷窃"（Steal Like an Artist）理念。

**与其他方法的区别**：

- 头脑风暴：从零开始，空白头脑
- 窃取风暴：已有素材为基础

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 14. Hanington & Martin《Universal Methods of Design》、15. Kleon《Steal Like an Artist》

### 6.2 操作流程

1. **收集素材**：搜集行业内外的相关案例、竞品功能、用户反馈
2. **建立素材库**：整理成可翻阅的卡片或文档
3. **随机抽取**：每次抽取1-3个案例作为"窃取源"
4. **强制嫁接**：思考"这个案例的哪个元素可以应用到我们的场景？"
5. **变异演化**：对原想法进行拼接、混合、变异、改造
6. **记录产出**
7. **重复迭代**

### 6.3 变异手段

- **拼接**：把A案例的A特点和B案例的B特点拼在一起
- **混合**：把两个想法融合成一个新的
- **夹杂**：把一个想法的元素穿插到另一个里
- **变异**：对原想法做夸张化、缩小化、反向化处理
- **演化**：保留核心思路，改变实现形式

### 6.4 示例

**议题：我们的电商APP如何创新？**

- **窃取来源一：美团外卖的实时骑手定位**

- **窃取来源二：游戏的签到连续奖励**

- **窃取来源三：微信的红包功能**

### 拼接想法

把美团外卖的"实时定位" + 电商结合：

- 包裹配送实时地图可视化，让用户看到"你的宝贝正在向你飞奔"
- 预测送达时间，精确到分钟

把游戏签到 + 电商结合：

- 连续签到解锁神秘福袋
- 签到天数越多，折扣越高（7天9折、30天7折）
- 漏签可以通过分享补卡机会

把微信红包 + 电商结合：

- 分享商品链接给好友，好友下单后双方都获得"分享红包"
- 直播间发红包，限时抢券
- 拼单成功后发起人获得"召集红包"

### 演化想法

将"实时定位"演化：

- 不仅看到包裹在哪，还看到"包裹的心情"——文案化表达："你的包裹正在穿越美丽的海岸线，预计下午3点与你相遇"

# 七、星爆式头脑风暴（Starbursting）

### 7.1 什么是星爆式头脑风暴

星爆式头脑风暴（Starbursting）是一种后期头脑风暴技术，在团队要实施具体方案前执行。从中心想法开始，围绕它创建一个六角星，六个角分别代表六个经典问题：谁（Who）、什么（What）、何时（When）、何地（Where）、为何（Why）、如何（How）。

**适合时机**：已有初步方案，需要全面审视时

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 6. Michalko《Thinkertoys》

### 7.2 六维度问题框架

| 维度 | 问题示例 |
|------|----------|
| Who（谁） | 谁会使用这个功能？谁是目标用户？谁会反对？谁受益最多？ |
| What（什么） | 这个功能具体做什么？它的边界是什么？什么情况下触发？ |
| When（何时） | 什么时候上线？什么时间节点发布？用户什么场景下使用？ |
| Where（何地） | 在哪里使用？APP哪个位置展示？什么环境下使用？ |
| Why（为何） | 用户为什么需要它？解决了什么痛点？为什么要现在做？ |
| How（如何） | 如何实现？用户如何使用？如何评价效果？ |

### 7.3 星爆图示意

```
               ┌──────────┐
               │  Who 谁  │
               └────┬─────┘
                    │
  ┌──────────┐      │      ┌──────────┐
  │ Why 为何 │──┐   │   ┌──│ What 什么│
  └──────────┘  │   │   │  └──────────┘
                ▼   ▼   ▼
              ┌─────────────┐
              │   中心想法  │
              └─────────────┘
                ▲   ▲   ▲
  ┌──────────┐  │   │   │  ┌──────────┐
  │ How 如何 │──┘   │   └──│Where 何地│
  └──────────┘      │      └──────────┘
                    │
               ┌────┴─────┐
               │ When 何时│
               └──────────┘
```

（六角发散：Who/What/When/Where/Why/How 围绕中心想法。）

### 7.4 示例

**中心想法：为APP添加AI个人助理功能**

### Who 谁

- **目标用户**：25-40岁的白领，需要日程管理、邮件处理、信息整理
- **受益方**：用户（效率提升）、公司（提高留存）
- **反对方**：担心隐私的用户
- **相关方**：客服（可能被AI替代部分工作）

### What 什么

- **功能**：AI帮你回复邮件、日程安排、总结会议内容、提醒待办
- **边界**：不做支付、不做社交、不做内容推荐
- **触发**：用户语音/文字输入指令

### When 何时

- **开发周期**：6个月
- **上线时间**：Q3
- **用户使用场景**：上班路上（42%）、午休（28%）、晚上（30%）

### Where 何地

- **展示位置**：首页下拉/悬浮按钮
- **使用环境**：地铁、办公室、家、任何有网络的地方
- **多端支持**：iOS/Android/Web

### Why 为何

- **痛点**：用户日程混乱、邮件太多看不过来、会议纪要整理费时间
- **市场机会**：市面上缺乏一站式AI效率工具
- **战略意义**：提高留存、增加付费转化

### How 如何

- **实现方式**：调用大模型API + 私有知识库
- **用户交互**：语音输入 + 文字输入
- **效果评估**：DAU增长5%、付费转化提升3%
- **迭代方式**：A/B测试，不断优化

# 八、人物角色风暴法（Figure Storming）

### 8.1 什么是人物角色风暴法

人物角色风暴（Figure Storming）是一种"社恐友好"的头脑风暴方法。设计团队设定一个不在房间里的知名人物（如乔布斯、马斯克、甄嬛），以这个人物的名义开始头脑风暴。参与者把自己的想法附在设定人物上，大胆提出创意。

**核心优势**：消除了"我提的想法好不好"的顾虑，因为这是"名人"的想法，不是"我"的想法。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 16. VanGundy《101 Activities for Teaching Creativity and Problem Solving》

### 8.2 操作流程

1. **选择角色**：选择一个与议题相关或无关的知名人物（可以跨领域、跨时代、跨次元）
2. **代入视角**：所有人以这个角色的视角、性格、价值观来看待问题
3. **自由发言**：以"如果是XX，他会觉得……"来表达想法
4. **记录汇总**
5. **复盘反思**：哪些想法真的来自这个角色的思维方式？

### 8.3 角色选择参考

| 角色类型 | 适合场景 |
|----------|----------|
| 乔布斯 | 产品设计、用户体验、极简主义 |
| 马斯克 | 颠覆式创新、第一性原理、大胆想法 |
| 甄嬛 | 宫廷剧/社交类APP、人际关系、权谋策略 |
| 爱因斯坦 | 科研类、技术突破、科学思维 |
| 马云 | 电商、商业模式、演讲文案 |
| 宫崎骏 | 创意设计、动画、游戏叙事 |

### 8.4 示例

**议题：我们的社交APP如何提升用户活跃度？**

### 设定角色：甄嬛

### 以甄嬛视角发言

**参与者A**（甄嬛模式）："皇上日理万机，若要臣妾们日日来请安，必要给些甜头才是。"

→ 启示：每日签到要给实质性好处，不能光给荣誉

**参与者B**（甄嬛模式）："华妃年家鼎盛，自然不把旁人放在眼里。若要众人服她，须得让她觉得与众不凡。"

→ 启示：要给活跃用户"身份感"，让他们觉得自己与众不同

**参与者C**（甄嬛模式）："臣妾听闻，皇上最忌结党营私，可若姐妹们情投意合，倒也不失为一段佳话。"

→ 启示：用户之间建立"小团体"，但要有健康的社交机制

**参与者D**（甄嬛模式）："皇后娘娘最在意规矩，可若处处都是规矩，臣妾们可就乏了。"

→ 启示：规则要适度，不能让用户感到被束缚

**参与者E**（甄嬛模式）："臣妾觉得，若能时常有些新鲜事、物，姐妹们自然愿意多来。"

→ 启示：持续更新内容/活动，让用户有新鲜感

### 复盘提炼

从甄嬛视角提炼可执行方案：

1. **等级权益系统**：不同等级对应不同"宫殿"身份
2. **姐妹圈/帮派系统**：用户可以组队，形成小团体归属感
3. **每日福利**：签到送实质权益（优惠券/会员体验）
4. **活动更新**：每周有新活动/话题，保持新鲜感
5. **仪式感设计**：升级有典礼、排行榜有展示

# 九、六顶思考帽（Six Thinking Hats）

### 9.1 什么是六顶思考帽

六顶思考帽（Six Thinking Hats）是"创新思维之父"爱德华·德·博诺（Edward de Bono）博士于1985年在《Six Thinking Hats》一书中提出的思维训练模式和全面思考问题的模型。

**核心概念**：提供"平行思维"的工具，避免将时间浪费在互相争执上。

**与传统思维的区别**：

- 传统思维：同时处理多种认知模式（混乱）
- 六顶思考帽：每次只处理一种认知模式（有序）

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 17. De Bono《Six Thinking Hats》、18.《Six Thinking Hats》- Wikipedia

### 9.2 六顶帽子详解

### 白帽｜信息与事实

**核心问题**：我们知道什么？我们需要知道什么？信息完整吗？

**关注点**：

- 事实、数据、已知的
- 信息缺口在哪里
- 需要补充什么信息

**发言格式**：

- "事实是……"
- "数据显示……"
- "我们需要的信息是……"

**白帽示例**

**议题：是否上线新产品X**

**白帽发言**：

- 我们的用户中，70%是25-35岁男性
- 上季度营收增长12%，但环比下降3%
- 竞品已上线类似功能，已运营6个月
- 我们的研发团队需要3个月开发

---

### 红帽｜情感与直觉

**核心问题**：我的感觉是什么？直觉告诉我什么？

**关注点**：

- 直觉、感受、预感
- 情绪、喜好
- 不用解释理由（感觉就是感觉）

**发言格式**：

- "我感觉……"
- "我的直觉告诉我……"
- "我觉得不对劲……"

**红帽示例**

**议题：是否上线新产品X**

**红帽发言**：

- "我感觉很兴奋，这个产品方向是对的"
- "我直觉觉得市场还没准备好"
- "我觉得这个项目会失败，但我不知道为什么"

---

### 黑帽｜批判与风险

**核心问题**：这会失败吗？风险是什么？逻辑漏洞在哪里？

**关注点**：

- 逻辑谬误
- 可能失败的原因
- 风险、障碍
- 与事实/经验不符的地方

**发言格式**：

- "这有一个风险……"
- "逻辑上有个问题……"
- "如果……会怎样？"

**黑帽示例**

**议题：是否上线新产品X**

**黑帽发言**：

- "我们的技术团队没有做AI的经验，3个月可能不够"
- "竞品已占先机，我们现在进入可能是红海"
- "如果用户不接受这个定价，销量会腰斩"

---

### 黄帽｜乐观与价值

**核心问题**：这有什么价值？最乐观的情况是什么？利益点在哪里？

**关注点**：

- 价值、利益
- 可行的理由
- 最乐观的预期
- 如何成功

**发言格式**：

- "优点是……"
- "这可以带来……"
- "如果成功的话……"

**黄帽示例**

**议题：是否上线新产品X**

**黄帽发言**：

- "这是我们进入AI赛道的最佳切入点"
- "如果拿下10%的目标市场，年收入增加5000万"
- "我们已有20万种子用户，可以快速冷启动"

---

### 绿帽｜创意与可能

**核心问题**：有什么新想法？有没有替代方案？有没有新途径？

**关注点**：

- 新点子
- 类比、跨越式想法
- 突破性创意
- "如果……会怎样"

**发言格式**：

- "我有一个想法……"
- "另一个可能是……"
- "如果我们改变前提……"

**绿帽示例**

**议题：是否上线新产品X**

**绿帽发言**：

- "我们可以先做免费版，积累用户再付费"
- "可以先和竞品合作，再慢慢超越"
- "可以做一个差异化功能，不正面竞争"

---

### 蓝帽｜流程控制与元思考

**核心问题**：我们应该怎么思考？我们下一步做什么？总结是什么？

**关注点**：

- 控制流程
- 总结、决定下一步
- 思考"我们在思考什么"
- 主持人的角色

**发言格式**：

- "让我们总结一下目前的讨论……"
- "下一步应该做什么……"
- "我们是否需要再讨论一下……"

**蓝帽示例**

**议题：是否上线新产品X**

**蓝帽发言**：

- "我们现在有了足够的信息，可以做决定了"
- "让我们投票决定是否进入开发阶段"
- "我们需要更多白帽信息（市场数据），暂停决策"

---

### 9.3 操作流程

### 顺序模式（适合一般讨论）

1. **蓝帽**：明确问题，设定议程
2. **白帽**：收集信息、事实
3. **绿帽**：提出新想法、新方案
4. **黄帽**：评估价值、好处
5. **黑帽**：批判风险、问题
6. **红帽**：直觉投票
7. **蓝帽**：总结决策

### 并行模式（适合复杂决策）

所有人同时戴上同一顶帽子，讨论完再换帽子。

### 个人模式（自我思考）

依次戴上每顶帽子，系统性思考问题的不同维度。

### 9.4 关键原则

1. **每次只戴一顶帽子**：不能同时处理多种认知模式
2. **帽子可以多次使用**：根据需要循环使用
3. **蓝帽不是中立的**：德·博诺特别强调，蓝帽是"元思考"——思考我们在思考什么
4. **红帽时间有限**：直觉判断要快速，不需要长篇大论
5. **黑帽要建设性**：批判是为了改进，不是为了否定

### 9.5 示例

**议题：公司是否应该全面采用远程办公？**

### 蓝帽（主持）

"好，我们今天讨论'是否全面采用远程办公'。我们依次用六顶帽子来思考。先请白帽发言。"

### 白帽（事实）

- 现有员工200人，80%支持远程
- 去年远程试点期间， Productivity 提升了15%
- 办公场地租金每年500万，可节省
- 沟通工具Slack、飞书、Zoom已部署

### 绿帽（新想法）

- 远程+定期线下结合（每月一次团建日）
- 建立"数字游民"文化，吸引全球人才
- 远程优先招聘，拓展人才库

### 黄帽（乐观）

- 节省500万租金，可用于员工福利
- 人才库扩大，招聘竞争力提升
- Productivity 提升15%，相当于增加30个人力

### 黑帽（风险）

- 远程办公可能导致团队凝聚力下降
- 部分员工自律性差，可能摸鱼
- 跨时区协作困难，可能影响沟通效率
- 核心技术资料泄密风险增加

### 红帽（直觉）

- "我觉得远程是未来，早转早受益"
- "我担心团队会散掉"
- "直觉告诉我混合模式最好"

### 蓝帽（总结）

"综合来看：支持远程的理由充分（省租金、提效率、扩人才），但需要配套文化建设和管理机制。建议采用'远程为主+每月线下日'的混合模式。先试点3个月再全面推广。"

# 十、德尔菲法（Delphi Method）

### 10.1 什么是德尔菲法

德尔菲法（Delphi Method）是一种结构化的专家判断预测方法，通过匿名多轮次专家调查，最终使意见收敛到一致或接近一致。

**来源**：该方法由兰德公司（RAND Corporation）的 Olaf Helmer、Norman Dalkey 与 Nicholas Rescher 在 1950年代初期的"Project Delphi"中开发，背景是冷战期间预测军事技术发展。名称源于希腊《德尔菲神谕》。代表性公开文献为 Dalkey & Helmer 1963年在《Management Science》上发表的"An Experimental Application of the Delphi Method to the Use of Experts"，使其广为人知。

**核心特征**：

- 匿名性：专家之间不直接接触，避免权威压力和群体思维
- 多轮反馈：每轮结束后整理统计结果，反馈给专家
- 收敛性：通过多次反馈，专家意见逐渐趋于一致

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 19. Dalkey & Helmer "An Experimental Application of the Delphi Method"、20.《Delphi Method》- Springer、21.《Delphi method》- Wikipedia

### 10.2 适用场景

- 需要专家判断，但无法召开实体会议
- 议题涉及多个领域，需要跨学科专家
- 问题不确定性强，缺乏历史数据
- 需要预测未来趋势或事件
- 专家分散在不同地区

### 10.3 标准操作流程

### 第一轮：提出问题

- 组织者确定议题，准备初始问卷
- 邀请专家（通常5-20人）
- 向专家发送问卷，请其独立回答

### 第二轮：汇总反馈

- 收集所有专家回答
- 整理、分类、统计
- 将汇总结果（不透露专家身份）反馈给所有专家
- 请专家参考他人意见后，重新回答

### 第三轮：再次反馈

- 统计第二轮结果
- 对于与群体差异大的专家，请其说明理由
- 再次反馈给全体专家

### 重复第N轮

- 持续进行直到：
  - 专家意见收敛到预定程度
  - 或达到预设轮数
  - 或专家明确表示不再改变意见

### 最终报告

- 编制最终报告
- 说明共识程度
- 列出主要分歧点（如果有）

### 10.4 问卷设计原则

1. **问题清晰**：避免模糊或歧义
2. **封闭式为主**：选择题、量表题为主
3. **允许开放式**：每轮留有开放式补充空间
4. **控制长度**：每轮问卷不超过20题
5. **提供背景**：附上必要的背景资料

### 10.5 收敛判断标准

通常采用以下指标判断收敛：

- 变异系数（CV）< 0.5
- 四分位距（IQR）较小
- 多数派比例 > 70%

### 10.6 示例

**议题：未来5年（2026-2031）中国新能源汽车市场份额预测**

### 第一轮问卷

向20位汽车行业专家发送问卷，问题包括：

1. 2031年新能源汽车（含混动）渗透率是多少？（填空：__%）
2. 影响渗透率的最主要因素是什么？（多选）
3. 您对预测的自信程度？（1-5分）

### 第一轮结果汇总

| 问题 | 中位数 | 范围 |
|------|--------|------|
| 2031年渗透率 | 68% | 45%-85% |

### 第二轮反馈

将第一轮汇总（不透露具体专家）反馈给所有专家，请其重新回答。

**调整示例**：

- 专家A原填80%，看到中位数68%后，调整为72%
- 专家B原填50%，但看到其他人普遍较高，上调至65%

### 第三轮结果

| 问题 | 中位数 | 范围 | 变异系数 |
|------|--------|------|----------|
| 2031年渗透率 | 71% | 60%-80% | 0.12 |

意见已收敛，停止调查，输出最终报告。

# 十一、世界咖啡（World Café）

### 11.1 什么是世界咖啡

世界咖啡（World Café）是一种通过轻松自然的对话产生集体智慧的汇谈方法。

**来源**：Brown, J. & Isaacs, D. (2005)《The World Café: Shaping Our Futures Through Conversations That Matter》

**核心理念**：像咖啡馆闲聊一样轻松，产生深度对话。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 22. Brown & Isaacs《The World Café》、23.《世界咖啡》- 机械工业出版社

### 11.2 环境布置

- **空间**：像咖啡馆一样布置，圆桌（4人/桌）
- **桌面**：铺上大纸或桌布，提供彩笔
- **氛围**：鲜花、烛光、轻音乐、点心
- **道具**：足够的便签纸、彩笔、计时器

### 11.3 标准流程

### 第一轮（20-45分钟）

1. 所有人分成4人一组，围绕1-2个对他们个人有意义的问题讨论
2. 有人做"桌长"，负责留守本桌；其他人是"使者"，可以到各桌流动
3. 使者每轮换桌，把上一桌的想法"带过去"并继续扩展

### 第二轮（20-45分钟）

1. 使者带着第一轮的洞见到新桌
2. 新桌在第一轮基础上继续深化
3. 重复以上流程

### 第三轮（可选）

继续轮换或合并成大组汇报

### 最终汇报

1. 所有桌长留在原位，汇报本桌的洞见
2. 全体参与者共同拼凑"大局图"

### 11.4 关键原则

1. **环境像咖啡馆**：轻松、温暖、没有压力
2. **问题对个人有意义**：不是抽象问题，而是与每个人相关的问题
3. **跨界混合**：尽可能让不同背景的人坐在一桌
4. **把想法写在桌上**：用大字写在桌布/大纸上
5. **混合重组**：通过使者机制，让想法不断扩展

### 11.5 示例

**议题：如何打造一个更有归属感的公司文化？**

### 第一轮（4人一组，共20分钟）

**A桌（程序媛们）**：

- 问题：什么让我觉得在公司有归属感？
- 产出：代码被尊重、技术成长、遇到问题有人帮

**B桌（产品经理们）**：

- 问题：什么让我觉得在公司有归属感？
- 产出：想法被听取、看到自己做的产品上线、有话语权

### 第二轮（使者带着第一轮洞见加入新桌）

**C桌（新混合组，包含来自A桌和B桌的使者）**：

- 新加入者带来A桌产出：技术成长很重要
- 新加入者带来B桌产出：话语权很重要
- 继续讨论：技术成长+话语权，如何结合？
- 产出：设立"技术委员会"，让工程师有决策权

### 第三轮 + 汇报

**各桌产出汇总**：

- A桌：代码评审文化、技术分享会
- B桌：产品功能委员会、用户反馈直达
- C桌：技术委员会、跨部门轮岗
- D桌：导师制、Buddy系统

**最终"大局图"**：
员工归属感 = 被尊重（话语权）+ 被成长（学习机会）+ 被连接（人际网络）

# 十二、苏格拉底式追问（Socratic Questioning）

### 12.1 什么是苏格拉底式追问

苏格拉底式追问是一种通过系统性提问来深化思考、揭露假设、检验逻辑的方法。

**来源**：柏拉图对话录，苏格拉底的"产婆术"

**核心思想**：通过提问而非告知，帮助对方自己发现真理。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 24. Paul & Elder《The Art of Socratic Questioning》、25.《Socratic method》- Wikipedia

### 12.2 六类追问框架

Richard Paul 与 Linda Elder 在《The Art of Socratic Questioning》中将苏格拉底式追问系统化为**六类**：

| 类型 | 问题示例 |
|------|----------|
| 澄清类 | "你的意思是……？" "能具体解释一下吗？" |
| 假设类 | "你的前提是什么？" "这个假设成立吗？" |
| 证据/推理类 | "证据在哪里？" "有什么数据支持？" "逻辑链是什么？" |
| 观点/视角类 | "背后的价值观是什么？" "谁受益？" "有没有其他观点？" |
| 影响/后果类 | "这样做会引发什么？" "短期/长期后果？" |
| 关于问题本身的问题 | "为什么这个问题重要？" "这个问题本身有没有问题？" |

### 12.3 操作要点

1. **连续追问**：不是一次提问，而是层层递进
2. **不预设答案**：追问是为了探索，不是为了证明对方错
3. **慢速**：给被问者足够的思考时间
4. **记录**：记录回答，以便后续追溯

### 12.4 示例

**对话场景：产品经理说"我们应该上这个功能，因为竞品也有"**

### 第一轮：澄清

**问**：竞品哪个有这个功能？

**答**：A公司和B公司。

**问**：具体是什么形式？

**答**：A公司是弹窗推送，B公司是站内信。

### 第二轮：假设

**问**：你的假设是什么？竞品上了，我们也应该上？

**答**：……

**问**：这个假设成立的前提是什么？

**答**：假设我们的用户和竞品用户需求一致。

### 第三轮：证据

**问**：有什么证据证明我们的用户需要这个功能？

**答**：用户调研里有5%的人提到。

**问**：5%的用户声音，这个量级足够吗？调研样本是多少？

### 第四轮：推理

**问**：从"5%用户提到"到"应该上线"，这个推理过程是什么？

**答**：需求存在，所以应该满足。

**问**：这个推理完整吗？需求存在→功能上线，这中间有没有遗漏步骤？

### 第五轮：观点

**问**：这个决策背后反映的是什么价值观？

**答**：不能落后于竞品。

**问**：还有谁会从这个决策中受益？

**答**：用户（如果有需求的话）、公司（功能完整）。

### 第六轮：影响

**问**：上了这个功能，有什么潜在负面影响？

**答**：可能打扰不需要的用户。

**问**：短期和长期影响分别是什么？

**答**：短期：可能提升部分用户体验。长期：可能流失不需要这个功能的用户。

### 第七轮：质疑

**问**：有没有其他解释或方案？

**答**：可以做成可关闭的选项，而不是默认开启。

**问**：有没有反例？有没有竞品不上这个功能也做得好的情况？

**答**：C公司就没上这个功能，但市场份额第一。

### 最终结果

通过追问，发现"竞品有所以我们要有"这个逻辑不严密。最终方案：做成可配置的开关功能，尊重用户选择。

# 十三、戈登分合法 / Synectics（提喻法）

### 13.1 什么是戈登分合法

戈登分合法（Synectics）是一种类比思考方法，由 William J.J. Gordon 与 George M. Prince 在 1950年代任职 Arthur D. Little 公司发明馆设计部门期间共同开发。Gordon 于 1961 年出版专著《Synectics: The Development of Creative Capacity》正式系统化阐述；Prince 于 1970 年出版《The Practice of Creativity》从便利者视角跟进。

**别名**：提喻法、综摄法、分合法

**核心思想**：通过同质异化使熟悉的事物变新奇，或通过异质同化使新奇的事物变熟悉。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 26. Gordon《Synectics》、27. Prince《The Practice of Creativity》

### 13.2 两种基本操作

### 同质异化（由合而分）

把熟悉的事物变得新奇——"如果这是全新的东西，我们会怎么做？"

### 异质同化（由分而合）

把陌生的事物变得熟悉——"这个问题像我们熟悉的什么？"

### 13.3 四种类比方法

### 直接类比

其他领域有没有类似情况？从中找灵感。

**示例**：航空领域 → 鸟翼 → 机翼设计

### 拟人类比

如果这是一个人，它会怎么想？有什么感受？

**示例**：如果这个APP是一个人，它是什么性格？
→ 它是个热心但有点啰嗦的管家

### 压缩冲突（象征类比）

将两个看似矛盾的概念压缩在一起，产生新洞见。

**示例**：

- "活的死物"→ 自愈材料
- "有序混乱"→ 敏捷开发中的Sprint

### 幻想类比

如果魔法可以实现，会怎样？打破物理限制。

**示例**：如果用户可以隔空操控APP，会是什么体验？

### 13.4 操作流程

1. **明确问题**：清晰陈述要解决的问题
2. **选择类比**：根据问题性质选择类比类型
3. **进行类比**：找到类比物，详细描述
4. **映射洞见**：把类比物的特性映射回原问题
5. **生成方案**：基于映射结果提出解决方案

### 13.5 示例

**问题：如何让用户更愿意在社交APP上发布内容？**

### 第一步：选择类比类型

选择"直接类比"——其他领域如何解决"让参与者更愿意行动"的问题？

### 第二步：研究类比物

**类比物：线下派对的主办方**

派对主办方如何让客人更愿意参与互动？

- 准备了好吃的食物（诱饵）
- 有互动游戏（降低门槛）
- 主人热情接待（情感连接）
- 介绍朋友给朋友（社交润滑）
- 有人带动气氛（KOL引导）

### 第三步：映射洞见

| 派对方法 | 映射到APP |
|----------|-----------|
| 好吃食物 | 发内容有奖励（积分/虚拟货币） |
| 互动游戏 | 降低发布门槛（模板/滤镜/音乐） |
| 主人热情接待 | 新手引导、欢迎消息 |
| 介绍朋友给朋友 | 社交匹配、@好友功能 |
| KOL带动 | 头部用户示范、官方活动 |

### 第四步：生成方案

1. 发布内容得积分，积分可兑换真实奖品
2. 提供"一键模板"，降低创作门槛
3. 新用户首周每日推送"发布邀请"
4. "老带新"机制，邀请朋友发布双方得奖励
5. 定期举办官方活动，提供流量扶持

# 十四、名义群体技术（NGT）

### 14.1 什么是名义群体技术

名义群体技术（Nominal Group Technique，NGT）是一种结构化的群体决策方法，在决策过程中对群体成员的讨论或人际沟通加以限制，但群体成员是独立思考的。

**来源**：André L. Delbecq 与 Andrew H. Van de Ven 于 1971 年在《Journal of Applied Behavioral Science》7(4): 466-492 发表论文 "A Group Process Model for Problem Identification and Program Planning" 正式提出；1975 年与 Gustafson 合著《Group Techniques for Program Planning: A Guide to Nominal Group and Delphi Processes》系统化阐述。

**核心特征**："名义"上是一个群体，实际上各自独立，防止少数人主导。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 28. Delbecq & Van de Ven "A Group Process Model"、29. Delbecq et al.《Group Techniques for Program Planning》、30.《Nominal Group Technique》- Wikipedia

### 14.2 适用场景

- 需要公平听取所有人意见
- 防止少数权威人士控制讨论
- 有明确决策选项，需要排序或选择
- 沉默者也需要发言机会
- 议题敏感，需要减少直接冲突

### 14.3 标准流程

### 步骤一：独立写想法（5-10分钟）

- 主持人宣布议题
- 每个人独立思考
- 在纸上写下自己的想法
- 不与他人讨论

### 步骤二：轮流陈述（10-15分钟）

- 主持人请每个人轮流简短陈述一个想法
- 每人每次只说一个想法
- 记录员写在白板/大纸上
- 不允许讨论或评价
- 循环直到所有想法都陈述完毕

### 步骤三：讨论澄清（10-15分钟）

- 所有人可以自由讨论
- 讨论仅限澄清想法含义
- 不允许辩论或说服
- 记录员根据讨论更新想法列表

### 步骤四：投票排序

- 每人匿名选择最重要的N个想法（如选5个）
- 给选中的想法排序（最重要=5分，第二=4分……）
- 汇总所有人的投票
- 得分最高的想法胜出

### 14.4 与头脑风暴的对比

| 维度 | 头脑风暴 | NGT |
|------|----------|-----|
| 讨论程度 | 高 | 低（名义上群体） |
| 匿名性 | 低 | 可高（投票环节） |
| 适合问题 | 开放性问题 | 有明确选项的决策 |
| 防止主导 | 一般 | 强 |
| 产出速度 | 快 | 慢但结构化 |

### 14.5 示例

**议题：团队下一季度最重要的一件事是什么？**

### 步骤一：独立写想法

每个团队成员独立写下3个想法：

- 成员A：上线新功能A、修复技术债、提升用户体验
- 成员B：提高留存率、降低成本、扩大用户群
- 成员C：优化推荐算法、开拓B端市场、提升客服质量
- 成员D：增加用户活跃度、加强安全、招聘人才

### 步骤二：轮流陈述

| 序号 | 想法 | 提出人 |
|------|------|--------|
| 1 | 上线新功能A | A |
| 2 | 修复技术债 | A |
| 3 | 提高留存率 | B |
| 4 | 降低成本 | B |
| 5 | 扩大用户群 | B |
| 6 | 优化推荐算法 | C |
| 7 | 开拓B端市场 | C |
| 8 | 提升客服质量 | C |
| 9 | 增加用户活跃度 | D |
| 10 | 加强安全 | D |
| 11 | 招聘人才 | D |

### 步骤三：讨论澄清

- 成员C解释"开拓B端市场"的具体含义
- 成员D澄清"加强安全"是指数据安全还是账号安全
- 其他成员补充问询，确保理解一致

### 步骤四：投票排序

每人选择最重要的5个，按重要性给5/4/3/2/1分：

| …… | …… |

### 最终结果

**第一名：上线新功能A**（16分）
→ 下一季度核心目标确定

# 十五、SCAMPER（奔驰法）

### 15.1 什么是SCAMPER

SCAMPER是一种改进现有产品/服务的创新方法，通过七个维度的提问，系统的激发新想法。

**来源**：基于奥斯本《Applied Imagination》中的"类比与激发"，由 Bob Eberle 整理成 SCAMPER。

**适用场景**：已有产品/服务，需要改进或寻找新功能。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 31. Eberle《SCAMPER》、32.《SCAMPER》- Wikipedia

### 15.2 七大策略详解

### S - Substitute（替代）

什么可以替代？换一个成分、材料、人、地点、时间、流程？

**追问**：

- 能否用其他成分替代？
- 能否用其他材料？
- 能否换个人来做这件事？
- 能否换个时间/地点？

### C - Combine（合并）

什么可以合并？跟其他东西组合在一起？

**追问**：

- 能否跟其他产品合并？
- 能否把两个功能合在一起？
- 能否合并用户群体？
- 能否合并流程？

### A - Adapt（改造）

怎样改造？改变形状、功能、颜色、味道、意义？

**追问**：

- 能否改变某些特征？
- 能否换个造型？
- 能否改变颜色？
- 能否调整尺寸？

### M - Modify（调整）

能否放大、缩小、夸张、加速、减速？

**追问**：

- 能否更大/更小？
- 能否更高/更矮？
- 能否更快/更慢？
- 能否更强/更弱？

### P - Put to other use（他用）

其他用途？换个使用场景、使用方式？

**追问**：

- 还能用在其他地方？
- 还能给其他人群用？
- 还能有其他使用方式？
- 还能有其他使用场景？

### E - Eliminate（去除）

能否去掉什么？简化、移除、拆分？

**追问**：

- 能否去掉某个部分？
- 能否简化？
- 能否拆分？
- 能否移除某个流程？

### R - Reverse / Rearrange（重组）

能否反过来？颠倒顺序、上下颠倒、正负交换？

**追问**：

- 能否反过来？
- 能否颠倒顺序？
- 能否交换正负？
- 能否重新排列？

### 15.3 示例

**产品：某款外卖APP**

### S（替代）

- 替代骑手配送 → 无人机/自动驾驶配送
- 替代纸质菜单 → AR菜单预览菜品
- 替代好评返现 → 好评解锁隐藏菜品

### C（合并）

- 外卖+社交 = 可以看到好友点了什么、推荐给好友
- 外卖+会员 = 会员专属餐厅、会员专属折扣
- 外卖+生鲜 = 食材配送+半成品食谱

### A（改造）

- 改造下单流程 = 社交下单（好友一起点）
- 改造评价体系 = 视频评价、语音评价
- 改造骑手服装 = 品牌联名骑手服

### M（调整）

- 放大配送速度 = 30分钟必达，慢了免单
- 缩小门槛 = 1元起送
- 调整营业时间 = 24小时营业

### P（他用）

- 外卖数据 → 餐饮商家选址参考
- 骑手网络 → 社区团购配送
- 评价系统 → 餐厅信用评级

### E（去除）

- 去除下单步骤中的确认页 → 一键下单
- 去除配送费 → 会员免费配送
- 去除评价字数限制 → 短视频评价

### R（重组）

- 预点餐→到店自取（顺序颠倒）
- 商家定价→用户定价（竞拍模式）
- 平台抽成→用户打赏（模式颠倒）

# 十六、KJ法（川喜田二郎法）

### 16.1 什么是KJ法

KJ法（川喜田二郎法）是由日本文化人类学者川喜田二郎（Jiro Kawakita）发明的，把大量事实、想法、语言素材卡片化，通过视觉化图解找出它们之间关联和规律的创意方法。

**核心思想**：将混沌的素材，通过"关联发现"变成有序的洞见。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 33. 川喜田二郎《発想法》、34. Scupin "The KJ Method"、35.《川喜田二郎》- Wikipedia

### 16.2 适用场景

- 探索性问题，前期调研
- 需求分析，用户访谈整理
- 大量反馈信息的归纳
- 跨部门沟通障碍打破
- 产品方向尚不明确时

### 16.3 操作流程

### 第一步：收集素材

- 实地调研、访谈、文献收集
- 不筛选，全部保留
- 素材可以是文字、数据、引用、感受

### 第二步：卡片化

- 将每条素材写在一张卡片上
- 一张卡片一个独立信息
- 标注信息来源

### 第三步：分类归纳

- 将卡片散放在桌上
- 寻找相似性、关联性
- 把相关的卡片放在一起
- 给每组命名一个标签

### 第四步：图解构建

- 将分类结果可视化
- 用箭头标注关系
- 找出核心主题
- 构建整体图

### 第五步：确立问题核心

- 从图中找出核心洞见
- 确定主要问题和次要问题
- 形成结论和建议

### 16.4 图解符号规范

| 符号 | 含义 |
|------|------|
| ○ | 原始素材卡片 |
| □ | 分类名称 |
| → | 因果关系/指向 |
| ⟷ | 相关关系 |
| ⊕ | 核心洞见 |

### 16.5 示例

**调研主题：用户为什么不满意我们的APP？**

### 第一步：收集素材（访谈10位流失用户）

- "界面太乱了，找不到想要的功能"（用户1）
- "打开速度太慢，等半天"（用户2）
- "客服回复太慢，解决问题要等3天"（用户3）
- "经常闪退，体验很差"（用户4）
- "推荐的东西不是我想要的"（用户5）
- "字体太小，看着费劲"（用户6）
- "注册流程太复杂"（用户7）
- "没有我想买的东西"（用户8）
- "支付方式太少了"（用户9）
- "卸载后重新安装很麻烦"（用户10）

### 第二步：卡片化

制作10张素材卡片

### 第三步：分类归纳

| 类别 | 卡片 |
|------|------|
| 性能问题 | 闪退、加载慢 |
| 界面设计 | 界面乱、字体小、找不到功能 |
| 推荐体验 | 推荐不准、没想要的东西 |
| 客服服务 | 回复慢 |
| 功能缺失 | 支付方式少、没有想要商品 |
| 注册/迁移 | 注册复杂、重新安装麻烦 |

### 第四步：图解构建

```
                    ┌─────────────────┐
                    │   用户不满体验   │
                    └────────┬────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
   │  产品体验   │    │  服务体验   │    │  技术体验   │
   │  (界面/推荐)│    │  (客服)     │    │  (性能)     │
   └──────┬──────┘    └──────┬──────┘    └──────┬──────┘
          │                  │                  │
    ┌─────┴─────┐            │            ┌────┴────┐
    ▼           ▼            ▼            ▼         ▼
  界面乱   推荐不准      回复慢        闪退    加载慢
  字体小   没想要商品
  找不到功能
```

### 第五步：核心洞见

**核心问题**：产品体验是用户流失的首要原因（4人），其中界面设计问题最突出。

**优先级排序**：

1. **P0**：技术性能（闪退/加载慢）——直接影响使用
2. **P1**：界面设计（乱/小/找不到）—— 影响第一印象
3. **P2**：推荐精准度——影响长期留存
4. **P3**：客服和功能缺失——锦上添花

# 十七、TRIZ（发明原理）

### 17.1 什么是TRIZ

TRIZ（俄语"发明问题解决理论"的首字母缩写，英译 Theory of Inventive Problem Solving）是由苏联发明家根里奇·阿奇舒勒（Genrich Altshuller）及其同事从 1946 年起开始发展。该理论建立在对全球专利文献中许多发明模式的系统研究上（原始研究样本在数万件级别，后续累计扩展至数十万至上百万件专利，不同资料口径不一）。

**核心发现**：发明问题中只有少数属于真正的突破式创新，绝大多数产业级创新问题在其他产业中已被解决过。

**核心理念**：用有限的40条原理来解决无限的发明问题。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 36. Altshuller《Creativity as an Exact Science》、37.《TRIZ理论——40个发明原理》、38.《TRIZ》- Wikipedia、39. TRIZ Portal

### 17.2 40个发明原理（精选详解）

### 原理1：分割

将物体分成独立部分。

- 例子：模块化手机、可拆卸家具

### 原理2：抽取

从物体中分离出干扰部分或唯一必要的部分。

- 例子：降噪耳机、隔离噪音

### 原理3：局部质量

让物体的不同部分有不同的功能。

- 例子：瑞士军刀、多功能笔

### 原理4：不对称

从对称形式转为非对称形式。

- 例子：人体工学鼠标、L型充电器

### 原理5：组合

将同类物体或时间上相邻的操作合并。

- 例子：智能手机（电话+电脑+相机）

### 原理6：多用性

一个物体执行多个功能，减少需要物体的数量。

- 例子：沙发床、充电宝+手电筒

### 原理7：嵌套

把一个物体放在另一个物体里面。

- 例子：套娃、折叠家具

### 原理8：配重

用轻的物体产生浮力或平衡。

- 例子：氢气球广告、无人机配重

### 原理9：预先反作用

预先施加必要的反作用。

- 例子：预应力混凝土

### 原理10：预先作用

预先完成必要的操作。

- 例子：快餐店预先切好蔬菜

### 17.3 矛盾矩阵

TRIZ将技术矛盾分为39个通用工程参数（如重量、速度、可靠性等），当两个参数冲突时，查矩阵找出推荐的发明原理。

**示例矛盾**：

- 强度 ↑ vs 重量 ↑ → 推荐原理：分割、嵌套、复合材料
- 速度 ↑ vs 能量消耗 ↑ → 推荐原理：相变、振动、一次性用品

### 17.4 物理矛盾分离法

当同一个参数需要同时满足两个相反的要求时，用分离法：

**空间分离**：在不同空间满足不同要求

- 例子：飞机机翼中部水平（减少阻力）+ 机翼根部垂直（承受压力）

**时间分离**：在不同时间满足不同要求

- 例子：折叠式头盔，戴着时透明，防晒时变成深色

**条件分离**：在不同条件下满足不同要求

- 例子：防水手机，干燥时正常扬声器，浸水时切换成骨传导

### 17.5 示例

**问题：如何让手机屏幕更抗摔？**

### 分析矛盾

**技术矛盾**：想要"强度"提升（改善），但代价是"重量"增加（恶化）。
（用 39 个工程参数定位：改善的是参数 14"强度"；恶化的是参数 1"运动物体的重量"。）

### 查矛盾矩阵

按 Altshuller 经典 39×39 矛盾矩阵（可在 triz40.com 在线查询），单元格 [改善14 × 恶化1] 推荐 4 条原理：

> **原理 1（分割）、原理 8（配重/反重量）、原理 40（复合材料）、原理 15（动态化）**

### 应用原理

**原理 1：分割**
将屏幕功能分成多个独立薄层而非整块厚玻璃。
→ 保护层（盖板玻璃）+ 显示层（OLED）+ 触控层（薄膜传感器），分别加工后贴合，整体厚度与重量可控。

**原理 40：复合材料**
用多种材料复合替代单一材料，兼顾硬度与韧性。
→ 蓝宝石涂层（高硬度）+ 康宁大猩猩玻璃（高韧性）+ 内层 PET 膜（缓冲），以复合层叠取代单层加厚。

**原理 15：动态化**
让结构在不同条件下呈现不同刚度，而非全程一种状态。
→ 在玻璃与中框之间引入弹性缓冲胶层（如硅胶/TPU），跌落瞬间发生弹性形变吸收冲击；屏幕悬浮于中框，受冲击时微位移分散应力。

（原理 8"配重"在手机场景较难直接应用，工程上常用前三条原理组合。）

### 最终方案

采用"复合三明治结构"：蓝宝石涂层 + 康宁玻璃 + PET 缓冲膜，玻璃与中框之间加弹性悬浮胶层，整机抗摔性显著提升而重量增幅可控。

# 十八、PMI 分析法

### 18.1 什么是PMI

PMI（Plus Minus Interesting）是由爱德华·德·博诺（Edward de Bono）提出的简单但强大的思考工具，是其 CoRT 思维工具（CoRT Thinking Lessons）系列的第一个工具，在1976年《Teaching Thinking》和1982年《de Bono's Thinking Course》中均有阐述。

**核心思想**：从一个想法的三个维度来审视——优点、缺点、兴趣点（值得进一步探索的地方）。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 40. De Bono《Teaching Thinking》/《de Bono's Thinking Course》、41. de Bono《CoRT Thinking Programme》

### 18.2 三维度详解

### Plus（优点）

这个想法有什么好处？有利因素是什么？

### Minus（缺点）

这个想法有什么问题？风险、代价是什么？

### Interesting（兴趣点）

这个想法中有什么有趣的、值得进一步探索的地方？可能是争议点、未知领域、或者激发新想法的点。

### 18.3 操作要点

1. **先不急着判断对错**：PMI不是打分，而是全面理解
2. **优点至少3条**：充分挖掘价值
3. **缺点要诚实**：不回避问题
4. **兴趣点最有价值**：往往是创新的来源
5. **按顺序来**：先P、再M、最后I

### 18.4 示例

**评估方案：全面推行996工作制**

### PMI评估

#### Plus 优点

- 项目交付速度加快
- 竞争对手也在996，不能落后
- 管理层期望这样

#### Minus 缺点

- 员工身心疲劳，效率下降
- 离职率上升，招聘成本增加
- 法律风险（违反劳动法）
- 团队氛围恶化

#### Interesting 兴趣点

- 如果只对核心项目组996，而不是全员？
- 如果"996"换成"弹性高强度"而不是固定时间？
- 能否通过提高工资来对冲员工不满？
- 竞争对手996的实际效果如何？有数据吗？

### 结论

从PMI分析看出：

1. 优点都是外部压力驱动，不是主动改善
2. 缺点严重且长期
3. 兴趣点指向"需要更精细化的方案"

**建议**：不全面推行996，而是对核心项目组采用"项目冲刺期"机制，冲刺期给足激励，非冲刺期正常双休。

# 十九、艾森豪威尔矩阵（Eisenhower Matrix）

### 19.1 什么是艾森豪威尔矩阵

**来源**：该理念源自美国第34任总统德怀特·D·艾森豪威尔在 1954 年于西北大学一场演讲中引述的名言：“我有两种问题，紧急的和重要的。紧急的不重要，重要的从不紧急。”后由 Stephen Covey 在1989年《高效能人士的七个习惯》（The 7 Habits of Highly Effective People）中将其系统化为四象限矩阵模型并广泛传播。

**核心思想**：事分轻重缓急，先做重要的事。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 42. Covey《The 7 Habits of Highly Effective People》、43.《Eisenhower Matrix》- Wikipedia、44.《Eisenhower Matrix》- Lifehack

### 19.2 四象限详解

### 第一象限：紧急且重要 → 立即执行

- 危机、紧急问题、有截止日期的任务
- **策略**：立刻做，不能拖延

### 第二象限：重要不紧急 → 计划执行

- 长期规划、预防措施、能力建设、关系建立
- **策略**：安排时间做，不要忽视

### 第三象限：紧急不重要 → 委托他人

- 某些会议、某些邮件、某些"别人紧急推过来的事"
- **策略**：尽量委托给别人做

### 第四象限：不紧急不重要 → 删除

- 刷手机、无意义的社交、纯娱乐
- **策略**：尽量减少或删除

### 19.3 操作步骤

1. **列出所有任务**
2. **评估每个任务**：
   - 它紧急吗？（时间上的紧迫性）
   - 它重要吗？（对目标/价值的贡献）
3. **归入四象限**
4. **按优先级执行**

### 19.4 示例

**产品经理的日常任务整理**

### 列出任务

1. 修复P0线上bug
2. 规划下季度产品路线图
3. 回复团队成员的问题
4. 参加跨部门协调会
5. 优化推荐算法（长期项目）
6. 参加行业协会论坛
7. 处理老板临时安排的任务
8. 整理用户反馈周报
9. 招聘面试
10. 刷行业新闻了解竞品动态

### 归入象限

| 象限 | 任务 |
|------|------|
| 第一象限（紧急且重要） | 1.修复P0 bug、7.老板临时任务、9.招聘面试 |
| 第二象限（重要不紧急） | 2.路线图规划、5.算法优化、8.用户反馈周报 |
| 第三象限（紧急不重要） | 3.回复团队问题、4.跨部门会议 |
| 第四象限（不紧急不重要） | 6.行业论坛、10.刷新闻 |

### 执行策略

- **上午**：处理P0 bug + 老板临时任务
- **下午**：做路线图规划（重要但不紧急）
- **委托**：团队问题请产品助理协调
- **删除**：刷新闻改为每天集中看15分钟

# 二十、思维导图法（Mind Mapping）

### 20.1 什么是思维导图

思维导图是一种将放射性思考可视化的方法，以一个核心概念为中心，向外发散出多个分支，每个分支代表与中心相关的一个想法，再继续向外延伸。

**创始人**：托尼·布赞（Tony Buzan），英国科普作家、教育顾问、电视节目主持人。1974年随BBC电视节目《Use Your Head》及同名配套书籍正式推广"mind map"这一术语和方法。布赞是"mind map"术语的提出者与系统化推广者，但放射状/树状笔记法此前已有数百年历史（如3世纪 Porphyry of Tyros、13世纪 Ramon Llull、达·芬奇笔记等）。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 45. Buzan《Use Your Head》、46.《Mind map》- Wikipedia、47. XMind 官网、48. MindNode 官网

### 20.2 思维导图规则

1. **中心主题**：放在中央，用图像或关键词
2. **主要分支**：从中心发散的粗线条，代表主要类别
3. **次要分支**：从主要分支延伸的细线，代表具体内容
4. **关键词**：每条线只写一个词，简短有力
5. **颜色**：不同分支用不同颜色，增强记忆
6. **图像**：适当加入小图标，增强视觉效果

### 20.3 推荐工具

- **XMind**：跨平台，功能全面
- **MindNode**：简洁优雅，适合Mac
- **ProcessOn**：在线协作
- **手绘**：白纸+彩笔，最自由

### 20.4 操作流程

1. **确定中心主题**：写在白纸中央，可以画一个简单的图
2. **发散主分支**：写下与中心相关的3-7个主要类别
3. **延伸次分支**：每个主分支继续向外延伸，写下相关想法
4. **添加细节**：继续扩展，层级不限
5. **回顾整理**：检查是否有遗漏，补充调整

### 20.5 示例

**主题：用户增长策略**

### 一级分支

- 渠道
- 内容
- 产品
- 用户
- 留存

### 二级分支（以"渠道"为例）

- 渠道→付费渠道→搜索引擎投放、信息流广告、KOL合作
- 渠道→免费渠道→SEO、社交媒体、内容营销
- 渠道→裂变→邀请奖励、拼团、分享得券

### 二级分支（以"留存"为例）

- 留存→会员体系→等级权益、专属福利、积分商城
- 留存→活动→签到、每日任务、节日活动
- 留存→社区→UGC内容、用户社群、话题讨论

# 二十一、莲花图法（Lotus Diagram）

### 21.1 什么是莲花图法

莲花图法（Lotus Blossom Technique / Lotus Diagram）是一种将中心主题延伸为八个（或多个）子主题，每个子主题再向外发散的思维工具，因形似莲花而得名。

**创始人**：日本管理顾问松村寧雄（Yasuo Matsumura，亦译"松村宁雄"）于 1980 年代前后开发，并通过其 MY 法（Matsumura's Yawaragi）系列著作推广。

**核心思想**：像花瓣一样层层展开，从核心到外围。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 54. 松村寧雄《MY 法ノート活用術》

### 21.2 结构

```
              子主题1
           ↗        ↖
      子主题8  中心主题  子主题2
           ↖        ↗
              子主题3
         ↗              ↖
      子主题7          子主题4
           ↖          ↗
              子主题5
         ↗              ↖
              子主题6
```

### 21.3 操作要点

1. **中心格**：写下核心主题
2. **第一层（8格）**：围绕中心的8个子主题
3. **第二层**：每个子主题再展开8个相关点
4. **层层递进**：根据需要继续延伸

### 21.4 示例

**主题：如何提升用户满意度**

### 中心格

用户满意度

### 8个子主题

1. **产品质量**——功能、性能、稳定性
2. **客户服务**——响应速度、专业度、态度
3. **价格价值**——定价合理、性价比
4. **用户体验**——界面、操作流程、文档
5. **品牌形象**——品牌信任、口碑
6. **创新迭代**——新功能、持续改进
7. **个性化**——定制服务、精准推荐
8. **可靠性**——稳定性、99.9% uptime

### 展开第二层（以"客户服务"为例）

- 响应速度→30秒内回复、7×24在线
- 专业度→知识库完善、首次解决率
- 态度→热情友好、主动跟进

# 二十二、曼陀罗思考法（Mandala Thinking）

### 22.1 什么是曼陀罗思考法

曼陀罗思考法是一种九宫格矩阵式扩展思考工具，以中心格 + 八个方向，形成一个3×3的网格。每个格子可以继续分裂成新的九宫格，形成无限延伸的思维网络。

**创始人**：日本设计师今泉浩晃（Hiroaki Imaizumi）于1987年前后提出，灵感来自佛教曼陀罗图形。

**核心思想**：全方位、多角度审视一个主题，不遗漏任何角度。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 55. 今泉浩晃《マンダラート発想法》

### 22.2 结构

```
┌─────────┬─────────┬─────────┐
│         │         │         │
│  方向8  │  方向1  │  方向2  │
│         │         │         │
├─────────┼─────────┼─────────┤
│         │         │         │
│  方向7  │ 中心格  │  方向3  │
│         │         │         │
├─────────┼─────────┼─────────┤
│         │         │         │
│  方向6  │  方向5  │  方向4  │
│         │         │         │
└─────────┴─────────┴─────────┘
```

### 22.3 与莲花图的区别

| 维度 | 曼陀罗 | 莲花图 |
|------|--------|--------|
| 形状 | 正方形九宫格 | 圆形花瓣 |
| 方向 | 直线向外延伸 | 环形发散 |
| 适合 | 结构化、系统性思考 | 探索性、开放性思考 |
| 层级 | 每个格子可继续分裂 | 每个花瓣可继续延展 |

### 22.4 示例

**主题：一次产品发布会**

### 第一层：3×3 曼陀罗

按上文 22.2 结构（中心顺时针：上=方向1，右上=方向2，右=方向3，右下=方向4，下=方向5，左下=方向6，左=方向7，左上=方向8）填充：

| 方向8：嘉宾邀请 | 方向1：场地布置 | 方向2：物料准备 |
|------|------|------|
| 方向7：内容策划 | 核心：发布会 | 方向3：宣传推广 |
| 方向6：彩排演练 | 方向5：执行统筹 | 方向4：媒体传播 |

### 第二层：嘉宾邀请（方向8）继续分裂

将上一层的"嘉宾邀请"放入新九宫格的**中心格**，再围绕它展开 8 个子项：

| 邀请函设计 | 邀请名单 | 确认跟进 |
|------------|------------|----------|
| KOL 名单 | **嘉宾邀请**（中心） | 交通住宿 |
| 媒体名单 | 现场座位 | 礼品准备 |

# 二十三、SWOT 矩阵法

### 23.1 什么是SWOT

SWOT是一种战略分析方法，通过四个维度（优势、劣势、机会、威胁）系统评估内外部环境。

**来源**：SWOT 的起源说法存在争议。主流考证指向两个 1965 年的独立源头：（1）斯坦福研究院（SRI）的 Robert F. Stewart、Otis J. Benepe 与 Arnold Mitchell 发表了使用“SOFT”（Satisfactory/Opportunities/Faults/Threats）的技术报告（后演变为 SWOT）；（2）哈佛商学院 Edmund P. Learned、C. Roland Christensen、Kenneth R. Andrews、William D. Guth 出版教材《Business Policy: Text and Cases》阐述了四要素内外部划分。Heinz Weihrich 于 1982 年发表的《The TOWS Matrix》提出了 2×2 矩阵。不少资料把 SWOT 归功于 Albert Humphrey，但该说法被 2023 年 Puyt 等发表的起源史研究质疑。

**适用场景**：战略规划、项目评估、竞争分析。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 49. Puyt et al. "The origins of SWOT analysis" / Weihrich "The TOWS Matrix"

### 23.2 四维度详解

### S - Strengths（优势）

内部积极因素：

- 我们的技术领先
- 品牌知名度高
- 团队执行力强
- 资金充裕

### W - Weaknesses（劣势）

内部消极因素：

- 产品线单一
- 人才流失
- 渠道覆盖不足
- 管理效率低

### O - Opportunities（机会）

外部积极因素：

- 市场需求增长
- 政策支持
- 新技术出现
- 竞争对手失误

### T - Threats（威胁）

外部消极因素：

- 市场饱和
- 经济下行
- 竞争对手崛起
- 法规变化

### 23.3 SWOT 组合策略

### SO策略（优势+机会）

利用优势抓住机会：

- 技术领先+市场需求增长→加大研发投入

### WO策略（劣势+机会）

利用机会弥补劣势：

- 产品线单一+新技术出现→开发新产品线

### ST策略（优势+威胁）

利用优势规避威胁：

- 品牌知名+竞争对手崛起→加强品牌建设

### WT策略（劣势+威胁）

弥补劣势规避威胁：

- 人才流失+市场不好→优化人才激励

### 23.4 示例

**分析：某咖啡品牌是否应该进入中国市场**

|  | 积极 | 消极 |
|--|------|------|
| **内部** | S（优势）：全球品牌影响力、成熟供应链、明星代言 | W（劣势）：对中国市场了解不足、本土团队弱 |
| **外部** | O（机会）：中国咖啡消费增长、年轻消费者崛起 | T（威胁）：已有瑞幸/星巴克、房租成本高、文化差异 |

### SWOT矩阵

| SO（进攻） | WO（改善） |
|------------|------------|
| 利用全球品牌+明星代言，快速建立知名度 | 组建本土团队，学习中国消费者习惯 |
| **ST（防御）** | **WT（防守）** |
| 与本土供应商合作，降低成本 | 差异化竞争，主打精品/特色路线 |

### 最终策略建议

**首选WT策略**（差异化竞争）：

- 主打"精品第三空间"，与瑞幸的"快咖啡"形成差异
- 先开1-2家旗舰店测试，积累经验后再扩张
- 与本土文化IP联名，降低文化壁垒

# 二十四、角色扮演法（Role Playing）

### 24.1 什么是角色扮演法

角色扮演法（Role Playing）让参与者代入不同角色（用户、对手、合作伙伴等）的视角来思考问题，发现自身盲点。

**来源**：角色扮演法源于 Jacob L. Moreno 在1920-30年代创立的心理剧（Psychodrama），后被广泛应用于决策、培训、教育等领域。

**核心价值**：打破"当事人视角"，看到问题的多面性。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 50. Moreno《Psychodrama, Volume 1》

### 24.2 常见角色类型

| 角色 | 视角价值 |
|------|----------|
| 终端用户 | 真实使用体验、痛点感知 |
| 一线员工 | 操作层困难、执行障碍 |
| 管理层 | 成本收益、风险控制 |
| 竞争对手 | 竞争策略、差异化方向 |
| 投资人 | 回报预期、退出路径 |
| 监管者 | 合规要求、潜在风险 |

### 24.3 操作流程

1. **选择角色**：根据议题选择代入的角色
2. **准备台词**：每个角色准备一份"台词"或"立场"
3. **演绎讨论**：角色之间进行模拟对话/辩论
4. **观察总结**：旁观者记录洞见
5. **复盘反思**：从角色扮演中提取可执行建议

### 24.4 示例

**议题：是否上线订阅制会员模式**

### 角色准备

**角色1：产品经理（支持方）**

"订阅制可以带来稳定的经常性收入（MRR），这是投资人最看重的指标。而且订阅用户有更高的LTV。"

**角色2：老用户代表（反对方）**

"我从免费时代就开始用了，现在要我付钱？我会流失到免费竞品那里去。"

**角色3：财务代表（谨慎方）**

"订阅制转型需要时间，短期收入可能下降。而且用户获取成本（CAC）会上升。"

**角色4：竞品CEO（外部视角）**

"我们暂时不会推订阅制，因为我们的用户群对付费很敏感。但如果我们做了，你们怎么做？"

### 演绎过程

**产品经理**：订阅制是SaaS化的必然趋势，我们可以提供专属功能……

**老用户**：但我不需要那些专属功能，为什么要我付费？

**财务**：如果流失30%用户，我们能承受吗？需要测算……

**竞品CEO**：你们先试，有效果我们再跟。

### 复盘洞见

1. 不能一刀切——应该做"Freemium"而不是纯付费
2. 老用户应该有优惠或过渡期
3. 财务需要做敏感性分析（流失率 vs 订阅率的平衡点）
4. 竞品在观望，我们先行有风险也有机会

# 二十五、Stefile 策略冲突法（自创）

### 25.1 什么是Stefile

Stefile（Strategy File）是一种让不同策略立场正面交锋的结构化辩论方法，通过模拟多个立场鲜明的角色，层层推进，最终识别最优解。

**适用场景**：策略选择、方案比选、重大决策。

📚 **本章参考文献**：见[附录2](#附录2参考文献与链接) 51.《设计思维方法集》- Stanford d.school、52.《Creative Problem Solving》- CPS、53.《行为设计学》- 微信读书

### 25.2 核心规则

1. **多角色同时存在**：每个角色代表一种策略立场
2. **顺序推进**：每轮必须基于上一轮结果推进，不许重复
3. **淘汰机制**：经过若干轮后，论点弱的角色被"淘汰"
4. **最终收敛**：最终剩下1-2个立场，进行最终比较

### 25.3 角色设定模板

| 角色 | 立场 | 核心论点 |
|------|------|----------|
| 激进派 | 大胆激进方案 | 高风险高回报 |
| 保守派 | 稳健渐进方案 | 低风险可控 |
| 创新派 | 颠覆式方案 | 弯道超车 |
| 整合派 | 组合方案 | 兼顾各方 |

### 25.4 操作流程

1. **设定议题**：明确要决策的问题
2. **设定角色**：为每个角色编写"角色卡"
3. **第一轮陈述**：各角色陈述核心立场
4. **第二轮攻击**：各角色针对其他角色的弱点发起攻击
5. **第三轮防守**：各角色回应攻击，补充论点
6. **淘汰投票**：其他"观察员"投票，最弱的角色淘汰
7. **重复1-2轮**：直到剩下最后立场

### 25.5 示例

**议题：公司是否应该ALL IN AI转型**

### 角色设定

**激进派（AI狂热者）**

"AI是下一次工业革命，不All in就会被淘汰。我们要成为AI公司，而不是'做AI的公司'。立即停止所有非AI项目，全部资源投入AI。"

**保守派（渐进改良者）**

"步子大了容易扯着蛋。应该用20%的资源试点，成功了再扩展。保持现有业务稳定是根本。"

**整合派（资源调配者）**

"两者不矛盾。可以把现有业务分ABC三类，A类业务加大AI投入，B类维持，C类逐步淘汰。资源重新配置，而不是简单 All in 或不变。"

### 第一轮陈述

**激进派**：AI是趋势，晚一步都是输……

**保守派**：我们有现金流业务，不能乱动……

**整合派**：两种声音都有道理，关键是资源怎么配比……

### 第二轮攻击

**激进派攻击保守派**：

"你说'试点'，但AI迭代速度这么快，等你试点完，市场早被抢光了！"

**保守派攻击激进派**：

"你说'All in'，万一AI方向错了呢？我们2022年才裁过一次员，不能再冒这个险。"

**整合派攻击两者**：

"你们都在非此即彼。激进派的All in风险太大，保守派的试点效率太低。我的方案才是最优解。"

### 第三轮防守

**激进派回应**：

"市场窗口就这1-2年，等我们试完，黄花菜都凉了。而且我有数据：早转型的公司存活率高出40%。"

**保守派回应**：

"你的数据样本有问题。那些'早转型'死的公司你没算进去。活着才是硬道理。"

**整合派回应**：

"我的方案结合了你们双方的优点——激进中有稳健，稳健中有激进。你们的方案都是走极端。"

### 淘汰投票（观察员投票，共 8 位观察员）

- 激进派：2票
- 保守派：1票
- **整合派：5票** → 进入最终对决

### 最终对决：整合派 vs 激进派

最终投票结果（同样 8 位观察员重新表决）：**整合派 6 票 : 激进派 2 票**，整合派胜出。

### 最终方案（整合派主张）

采用"A类加速、B类维持、C类淘汰"的资源重配策略：

- A类（占40%资源）：AI优先，ALL IN
- B类（占50%资源）：稳步推进，小幅AI化
- C类（占10%资源）：逐步收缩，2年内退出

### 方法组合使用示例

以下 8 套组合覆盖产品、战略、远程协作、教学、应急、个人决策、研发、用户研究 8 类典型场景。每套组合都给出一个具体的应用实例，便于参考。

#### 组合一：产品创新完整工作坊（2 小时，5-12 人）

| 阶段 | 时长 | 方法 |
|------|------|------|
| 1. 热身破冰 | 10 min | 游戏化风暴法（Gamestorming） |
| 2. 发散 | 30 min | 经典头脑风暴 |
| 3. 视角分流 | 20 min | 六顶思考帽（初筛） |
| 4. 深化改造 | 30 min | SCAMPER（奔驰法） |
| 5. 多面评估 | 20 min | PMI 分析法 |
| 6. 决策投票 | 10 min | NGT 名义群体技术 |

**应用实例：为一款健身 App 新增"AI 私教"功能**

#### 组合七：研发技术攻关工作坊（4 小时，3-8 名工程师）

| 5. 方案评估 | 30 min | PMI 分析法 |
| 6. 决策 | 15 min | NGT 投票 |

**应用实例：让一款边缘设备的视觉模型推理延迟从 200 ms 降到 50 ms**

星爆式头脑风暴在第一阶段就把"50 ms"约束、"端侧""实时""测试环境"等条件写明，避免方案选完才发现测不出。TRIZ 用来解"更高精度↔更低延迟"的矛盾，可参考参数变化（原理 35）和分割（原理 1，对应模型蒸馏与级联）。Synectics 引入"人眼中央凹高分辨率、周边低分辨率"的类比，启发出"ROI 检测 + 局部高精度推理"的思路。窃取风暴法则用于整合 MobileNet、TensorRT 与旧项目的剪枝脚本，PMI 进一步评估各候选方案的优、缺与有趣点，避免只看单一 benchmark。整个流程覆盖从问题定义、矛盾抽象、灵感、缝合到落地的完整研发链路。

#### 组合八：用户需求挖掘工作坊（3 小时，4-10 名产品/UX）

| 阶段 | 时长 | 方法 |
|------|------|------|
| 1. 用户共情 | 40 min | 角色扮演法（扮演 3 类典型用户走流程） |
| 2. 痛点收集 | 30 min | 脑力书写 6-3-5（每人写 3 个痛点 × 5 轮） |
| 3. 痛点归纳 | 40 min | KJ 法（亲和图分组） |
| 4. 反向追问 | 30 min | 质疑头脑风暴 + 苏格拉底追问（"为什么这个痛点存在？"） |
| 5. 强制联想 | 30 min | 触发因子头脑风暴（用随机词刺激解决方案） |
| 6. 优先级排序 | 10 min | 艾森豪威尔矩阵（用户价值 × 实现难度变体） |

**应用实例：在线教育平台为"二三线城市家长"挖掘新功能**

角色扮演覆盖三类典型家长（双职工、全职妈妈、留守祖辈），暴露真实使用情境；KJ 法把 50 多个原始痛点收敛为 6-8 个亲和簇。质疑头脑风暴与苏格拉底追问深挖根因——例如"为什么家长不点开学习报告？"，往往会从"按钮太小"这种表层解释回到"看不懂术语"这一真问题；触发因子头脑风暴再用"快递柜""短视频"等随机词刺激解决方案，启发出"学习报告短视频化"等创意。最后用艾森豪威尔矩阵的变体（用户价值 × 实现难度）排序，确保产出的不是"团队觉得酷"，而是"用户真的会用"。这套组合的"反向追问 + 强制联想"专门用于穿透伪需求。

### 决策树自检清单

使用决策树前，建议先快速核对以下五点：

1. 是否能用一句话写出此次讨论**唯一**的主目标？如果不能，应拆分成多次会议或直接选用组合。
2. 约束是否清楚：人数、时长、是否需要匿名、是否远程？
3. 期望产出是哪一类：发散点子清单、收敛 Top N、单一决策、还是行动计划？
4. 是否为每位参与者匹配了"安全的发声方式"——社恐者有书面通道、专家有匿名通道？
5. 是否准备了收口方法（NGT、Stefile 或德尔菲终轮），而不是只发散不收敛？
